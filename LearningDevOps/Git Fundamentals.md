## Краткое объяснение

Для DevOps-инженера Git — это не просто хранилище кода, а фундамент подхода **Infrastructure as Code (IaC)**. Мы используем его, чтобы версионировать конфигурации серверов, скрипты и пайплайны. Это «единый источник истины»: если изменений нет в Git, значит, их нет в инфраструктуре.

> [!note] Определение
> **Git** — это распределенная система контроля версий. Она позволяет нескольким людям работать над одним кодом, хранит историю изменений и позволяет откатываться к любой точке в прошлом.

## Ключевые понятия темы

- **Репозиторий**: база данных, хранящая историю изменений проекта.
- **Commit (Коммит)**: «снимок» состояния файлов в определенный момент времени.
- **Stage (Индекс/Index)**: промежуточная зона, куда файлы добавляются перед коммитом.
- **Branch (Ветка)**: изолированная линия разработки.
- **Blob, Tree, Commit, Tag**: четыре базовые сущности внутреннего устройства Git.
## Внутренние объекты Git
- **Blob**: Содержимое конкретного файла.
- **Tree**: Структура каталогов (ссылки на блобы и другие деревья).
- **Commit**: Ссылка на дерево, данные об авторе и родителе.

> [!example] Основной цикл работы
> 1. `git status` — проверка состояния файлов.
> 2. `git add <file>` — добавление в индекс (Stage).
> 3. `git commit -m "сообщение"` — фиксация изменений.
> 4. `git log` — просмотр истории коммитов.

## Практика (минимальный рабочий пример)

Создадим локальный репозиторий для твоих скриптов:
```
# Установи git, если его еще нет
sudo apt update && sudo apt install git -y

# Настрой имя и почту (обязательно для истории коммитов)
git config --global user.name "Your Name"
git config --global user.email "your@email.com"

# Инициализируй репозиторий в папке со скриптами
cd ~/projects/learning/linux
git init

# Проверь статус (увидишь свои .sh файлы как untracked)
git status

# Добавь файл в индекс и закоммить
git add service_manager.sh
git commit -m "feat: add service manager script"

# Посмотри историю
git log --oneline
```

## Задание

**Цель**: Научиться фиксировать изменения и работать с историей. **Шаги**:

1. Добавь в свой скрипт `service_manager.sh` комментарий в первую строку (после шебанга).
2. Посмотри разницу между текущим файлом и последним коммитом через `git diff`.
3. Закоммить это изменение с описанием: `docs: add comment to script`.
4. Создай новую ветку `feature/test-branch` командой `git checkout -b feature/test-branch`.
5. Создай в этой ветке пустой файл `README.md`, закоммить его, а потом вернись в ветку `main` (или `master`) командой `git checkout main`. **Ожидаемый результат**: Понимание, как Git переключает состояние файлов. После перехода в `main` файл `README.md` должен исчезнуть из папки. **Как проверить себя**: Команда `git branch` покажет список веток и ту, на которой ты сейчас находишься
