## Краткое объяснение

Для DevOps-инженера Git — это не просто хранилище кода, а фундамент подхода **Infrastructure as Code (IaC)**. Мы используем его, чтобы версионировать конфигурации серверов, скрипты и пайплайны. Это «единый источник истины»: если изменений нет в Git, значит, их нет в инфраструктуре.

> [!note] Определение
> **Git** — это распределенная система контроля версий. Она позволяет нескольким людям работать над одним кодом, хранит историю изменений и позволяет откатываться к любой точке в прошлом.

## Ключевые понятия темы

- **Репозиторий**: база данных, хранящая историю изменений проекта.
- **Commit (Коммит)**: «снимок» состояния файлов в определенный момент времени.
- **Stage (Индекс/Index)**: промежуточная зона, куда файлы добавляются перед коммитом.
- **Branch (Ветка)**: изолированная линия разработки.
- **Blob, Tree, Commit, Tag**: четыре базовые сущности внутреннего устройства Git.
## Внутренние объекты Git
- **Blob**: Содержимое конкретного файла.
- **Tree**: Структура каталогов (ссылки на блобы и другие деревья).
- **Commit**: Ссылка на дерево, данные об авторе и родителе.

> [!example] Основной цикл работы
> 1. `git status` — проверка состояния файлов.
> 2. `git add <file>` — добавление в индекс (Stage).
> 3. `git commit -m "сообщение"` — фиксация изменений.
> 4. `git log` — просмотр истории коммитов.
