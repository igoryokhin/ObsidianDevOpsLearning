## Краткое объяснение

	В продакшене **Nginx** редко используется просто для отдачи статических файлов. Его главная роль — **Reverse Proxy** (обратный прокси). Он принимает запросы из интернета и перенаправляет их на внутренние сервисы (например, на приложение на Python или Node.js), скрывая их реальные IP и добавляя уровень безопасности и балансировки.

> [!note] Reverse Proxy
> **Обратный прокси** — это сервер, который стоит между клиентом и бэкендом. Он принимает запросы и пересылает их на нужный сервис.

## Ключевые понятия темы

- **Reverse Proxy**: сервер, который перенаправляет запросы клиентов на один или несколько бэкенд-серверов.
- **proxy_pass**: основная директива Nginx для перенаправления трафика.
- **Location**: блок в конфиге, определяющий, как обрабатывать конкретный URL-путь.
- **Upstream**: группа серверов, между которыми Nginx может балансировать нагрузку
## Ключевые директивы
- `proxy_pass` — адрес, куда переслать запрос.
- `proxy_set_header` — передать оригинальные заголовки клиента бэкенду.
- `upstream` — описание группы серверов для балансировки.

> [!example] Пример простого прокси
> ```nginx
> location /app {
>     proxy_pass http://127.0.0.1:3000;
>     proxy_set_header Host $host;
> }
> ```

## Практика (минимальный рабочий пример)
Давай настроим Nginx так, чтобы при обращении к нему он "проксировал" запрос на какой-нибудь внешний сайт (эмуляция бэкенда). Отредактируй стандартный конфиг `/etc/nginx/sites-available/default` (через `sudo nano`):

```
server {
    listen 80;
    server_name myserver.local;

    location /google {
        proxy_pass http://google.com/;
    }
}
```

### Чек-лист
- [ ] Конфиг проверен через `nginx -t`.
- [ ] Создана ссылка в `sites-enabled`.
- [ ] Сервис перезапущен через `systemctl`.
